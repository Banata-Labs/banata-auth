This is the standard Banata integration path.

You do **not** embed Banata's auth backend in your app. You:

1. create a project in the Banata dashboard
2. configure auth methods and providers for that project
3. create a project-scoped API key
4. proxy `/api/auth` through your app
5. render auth UI inside your own frontend

If you self-host Banata, this flow stays the same. Only the Banata URLs change.

---

## Dashboard Prerequisites

Before you write code:

1. Sign in to the Banata dashboard.
2. Open or create the project you want your app to use.
3. Enable the auth methods you want under **Authentication -> Methods**.
4. Configure provider credentials under **Authentication -> Providers**.
5. If you enable email-based auth, configure **Emails** and templates.
6. Create a **project-scoped API key** under **API Keys**.

If you skip steps 3 to 5, your frontend can still render forms, but the backend will reject flows that are not enabled for that project.

## Install Packages

```bash
npm install @banata-auth/nextjs @banata-auth/react @banata-auth/sdk
```

Use:

- `@banata-auth/nextjs` for the auth route proxy and server helpers
- `@banata-auth/react` for sign-in and sign-up UI in your app
- `@banata-auth/sdk` when you want to configure Banata by code

## Environment Variables

```bash
NEXT_PUBLIC_SITE_URL=http://localhost:3000

NEXT_PUBLIC_CONVEX_URL=https://your-banata-instance.convex.cloud
NEXT_PUBLIC_CONVEX_SITE_URL=https://your-banata-instance.convex.site

BANATA_API_KEY=sk_test_your_project_key
```

Notes:

- `BANATA_API_KEY` must stay server-side.
- `NEXT_PUBLIC_CONVEX_SITE_URL` is the Banata auth backend URL.
- `NEXT_PUBLIC_CONVEX_URL` is used by server-side Convex helpers.

## Create Server Auth Helpers

```ts
// src/lib/auth-server.ts
import { createBanataAuthServer } from "@banata-auth/nextjs/server";

function requireEnv(name: string): string {
  const value = process.env[name];
  if (!value) {
    throw new Error(`${name} is required.`);
  }
  return value;
}

export const {
  handler,
  isAuthenticated,
  getToken,
  preloadAuthQuery,
  fetchAuthQuery,
  fetchAuthMutation,
  fetchAuthAction,
} = createBanataAuthServer({
  convexUrl: requireEnv("NEXT_PUBLIC_CONVEX_URL"),
  convexSiteUrl: requireEnv("NEXT_PUBLIC_CONVEX_SITE_URL"),
  apiKey: requireEnv("BANATA_API_KEY"),
});
```

The API key is what binds this app to one Banata project.

## Create The Auth Route

```ts
// src/app/api/auth/[...all]/route.ts
import { handler } from "@/lib/auth-server";

export const { GET, POST, PUT, PATCH, DELETE, OPTIONS } = handler;
```

Your browser should call `/api/auth` on your own app origin, not the Banata backend URL directly.

## Create The Browser Auth Client

```ts
// src/lib/auth-client.ts
"use client";

import { createAuthClient } from "@banata-auth/react/plugins";

export const authClient = createAuthClient({
  baseURL: "/api/auth",
});
```

## Add Sign-In And Sign-Up Pages

```tsx
// src/app/sign-in/page.tsx
"use client";

import Link from "next/link";
import { SignInForm } from "@banata-auth/react";
import { authClient } from "@/lib/auth-client";

export default function SignInPage() {
  return (
    <SignInForm
      authClient={authClient}
      callbackURL="/"
      title="Sign in"
      description="Use the methods enabled for this Banata project."
      footer={<Link href="/sign-up">Create an account</Link>}
    />
  );
}
```

```tsx
// src/app/sign-up/page.tsx
"use client";

import Link from "next/link";
import { SignUpForm } from "@banata-auth/react";
import { authClient } from "@/lib/auth-client";

export default function SignUpPage() {
  return (
    <SignUpForm
      authClient={authClient}
      callbackURL="/"
      title="Create account"
      description="This project is powered by Banata."
      footer={<Link href="/sign-in">Already have an account?</Link>}
    />
  );
}
```

## Guard Your App

```tsx
// src/app/page.tsx
import { redirect } from "next/navigation";
import { isAuthenticated } from "@/lib/auth-server";

export default async function HomePage() {
  const signedIn = await isAuthenticated();

  if (!signedIn) {
    redirect("/sign-in");
  }

  return <div>Authenticated app shell</div>;
}
```

## Configure Banata By Code

The dashboard is the main management surface, but it is not the only way to configure a project.

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});

await banata.configuration.saveDashboardConfig({
  authMethods: {
    emailPassword: true,
    emailOtp: true,
  },
  emailPassword: {
    requireEmailVerification: true,
    autoSignIn: true,
  },
});

await banata.configuration.saveBrandingConfig({
  primaryColor: "#0f766e",
  bgColor: "#f5f5f4",
  borderRadius: 14,
});
```

These writes update the same project config that the dashboard reads. If you change auth settings by code, the dashboard reflects those changes immediately.

## What To Verify

After wiring the app:

1. Open `/sign-up` or `/sign-in` in your app.
2. Create an account or sign in.
3. Confirm `/api/auth/get-session` returns a live session.
4. Confirm the same user appears in the Banata dashboard under **Users**.
5. Confirm project configuration changes in the dashboard affect the app without local auth rewiring.

## Common First-Run Problems

### The frontend renders, but sign-in fails immediately

Check:

1. the project has the matching auth method enabled
2. the required provider credentials are configured
3. the app is using the correct `BANATA_API_KEY`

### Email sign-up exists, but the flow is not practical

Email delivery is not ready yet. Configure:

1. an email provider
2. the relevant templates
3. the email policy you want for that project

### Sign-in succeeds, but the app still has no session

Check:

1. your browser is calling `/api/auth`, not the Banata backend directly
2. the app and auth route are on the same origin
3. `/api/auth/get-session` works in your browser session
4. the project API key is valid and scoped correctly

### Repeated first-run testing starts hitting rate limits

Banata rate-limits sensitive auth routes. That protection is project-aware and, for email-based endpoints, identifier-aware, but repeated tests from one shared IP can still hit the limit window.

If that happens:

1. wait for the retry window to expire
2. use a fresh email when testing sign-up
3. avoid repeatedly submitting the same hosted form while you are debugging app-side routing
