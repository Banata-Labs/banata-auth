Banata should be understood in two layers:

1. **Your app integration**
2. **The Banata platform**

Most customers only need the first layer.

---

## What Lives In Your App

For the normal integration path, your app only needs a small auth surface:

| File or package | Purpose |
|---|---|
| `@banata-auth/nextjs` | Route proxy and server auth helpers |
| `@banata-auth/react` | Sign-in/sign-up UI, provider, and auth hooks |
| `@banata-auth/sdk` | Programmatic configuration and admin APIs |
| `src/lib/auth-server.ts` | Server-side Banata binding |
| `src/lib/auth-client.ts` | Browser client for `/api/auth` |
| `src/app/api/auth/[...all]/route.ts` | Same-origin auth route in your app |

Your app owns the application shell and the route surface. Banata owns the auth data and the runtime behind that route.

---

## What Lives In Banata

Banata owns these concerns per project:

- users and sessions
- social accounts
- organizations and invitations
- roles, permissions, and resource types
- auth methods and provider credentials
- email templates and delivery configuration
- project branding and redirect configuration
- audit logs, webhooks, SSO, and SCIM

---

## Typical App Structure

```text
src/
├── app/
│   ├── api/auth/[...all]/route.ts
│   ├── sign-in/page.tsx
│   ├── sign-up/page.tsx
│   └── page.tsx
└── lib/
    ├── auth-client.ts
    └── auth-server.ts
```

That is the core of the customer integration. Everything else is normal app code.

---

## What You Do Not Add

For the normal Banata integration path, you do **not** add:

- a local auth database schema
- a local Better Auth runtime
- a local `convex/banataAuth` runtime inside your product app

Those belong to the Banata platform, not to each customer app.

---

## Self-Hosting Changes Infrastructure, Not The Customer API

If you self-host Banata, you deploy the Banata platform itself. Customer apps still:

1. sign in to a Banata dashboard
2. create projects
3. create API keys
4. connect through the same SDKs and route helpers

The main difference is that the Banata URLs point to your self-hosted deployment instead of Banata's hosted cloud.
