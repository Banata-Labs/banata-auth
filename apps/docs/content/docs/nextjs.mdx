`@banata-auth/nextjs` is the main server-side package for customer apps.

Use it to:

- proxy `/api/auth/*` through your app
- keep cookies on your own domain
- bind the app to one Banata project with a server-side API key
- read auth state in server components, actions, and route handlers

```bash
npm install @banata-auth/nextjs
```

---

## Recommended Model

1. Create a project in the Banata dashboard.
2. Configure auth methods, providers, and emails for that project.
3. Create a project-scoped API key.
4. Store the key in your server environment.
5. Proxy `/api/auth` through your app.
6. Render sign-in and sign-up UI in your own frontend.

The browser never sees the API key.

---

## Server Helpers

```ts
// src/lib/auth-server.ts
import { createBanataAuthServer } from "@banata-auth/nextjs/server";

export const {
  handler,
  isAuthenticated,
  getToken,
  preloadAuthQuery,
  fetchAuthQuery,
  fetchAuthMutation,
  fetchAuthAction,
} = createBanataAuthServer({
  convexUrl: process.env.NEXT_PUBLIC_CONVEX_URL!,
  convexSiteUrl: process.env.NEXT_PUBLIC_CONVEX_SITE_URL!,
  apiKey: process.env.BANATA_API_KEY!,
});
```

This uses the same API key for the auth route, token refreshes, and server-side auth helpers.

---

## Route File

```ts
// src/app/api/auth/[...all]/route.ts
import { handler } from "@/lib/auth-server";

export const { GET, POST, PUT, PATCH, DELETE, OPTIONS } = handler;
```

Your app should talk to `/api/auth` on its own origin. Do not call the Banata backend URL directly from the browser.

---

## Read Auth State On The Server

```tsx
import { redirect } from "next/navigation";
import { isAuthenticated } from "@/lib/auth-server";

export default async function DashboardPage() {
  const signedIn = await isAuthenticated();

  if (!signedIn) {
    redirect("/sign-in");
  }

  return <div>Authenticated dashboard</div>;
}
```

## Environment Variables

```bash
NEXT_PUBLIC_SITE_URL=http://localhost:3000
NEXT_PUBLIC_CONVEX_URL=https://your-banata-instance.convex.cloud
NEXT_PUBLIC_CONVEX_SITE_URL=https://your-banata-instance.convex.site

BANATA_API_KEY=sk_test_your_project_key
```

`BANATA_API_KEY` must stay server-side.

## Programmatic Configuration

`@banata-auth/nextjs` handles transport and cookies. If you want to change the project's Banata configuration by code, use the SDK:

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});

await banata.configuration.saveDashboardConfig({
  authMethods: {
    emailPassword: true,
    emailOtp: true,
  },
  emailPassword: {
    requireEmailVerification: false,
    autoSignIn: true,
  },
});
```

That updates the same project config the dashboard reads.

## Troubleshooting

### The frontend can render, but auth calls fail

Check:

1. the app is calling `/api/auth`
2. the project API key is valid
3. the target Banata project has the required methods enabled

### The app has no session after sign-in

Check:

1. the auth route is mounted inside your app
2. the app and auth route are on the same origin
3. `/api/auth/get-session` works in the browser session

### Do I still need `projectId` or `clientId` in app code?

Normally no.

The API key is the main project-binding mechanism for the app backend.
