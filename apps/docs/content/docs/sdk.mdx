`@banata-auth/sdk` is the admin and configuration client for Banata.

Use it when your backend needs to:

- manage users, organizations, invitations, SSO, SCIM, and RBAC
- manage API keys and webhooks
- update auth, branding, and provider configuration by code
- keep code-driven configuration and dashboard configuration in sync

```bash
npm install @banata-auth/sdk
```

---

## Initialization

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});
```

Start by creating the project and API key in the Banata dashboard. Then use that project-scoped key in the SDK.

---

## Dashboard And Code Stay In Sync

Banata is dashboard-first, but configuration can also be updated by code.

There is no separate dashboard config store and SDK config store.

That means you can:

- change auth methods from code
- update provider credentials from code
- update branding from code
- see those same values in the dashboard immediately

---

## Configure Auth Methods By Code

```ts
await banata.configuration.saveDashboardConfig({
  authMethods: {
    emailPassword: true,
    emailOtp: true,
    magicLink: false,
    passkey: false,
  },
  emailPassword: {
    requireEmailVerification: true,
    autoSignIn: true,
    minPasswordLength: 10,
    maxPasswordLength: 128,
  },
});
```

This updates the same project config the dashboard reads under **Authentication**.

---

## Configure Branding By Code

```ts
await banata.configuration.saveBrandingConfig({
  primaryColor: "#0f766e",
  bgColor: "#f5f5f4",
  borderRadius: 14,
  font: "General Sans",
  logoUrl: "https://example.com/logo.svg",
});
```

---

## Configure Social Providers By Code

```ts
await banata.configuration.saveSocialProviderCredential({
  providerId: "github",
  clientId: process.env.GITHUB_CLIENT_ID!,
  clientSecret: process.env.GITHUB_CLIENT_SECRET!,
  enabled: true,
});
```

That writes to the same provider configuration surfaced in the dashboard.

## Example Admin Workflow

```ts
const role = await banata.rbac.createRole({
  name: "HR Admin",
  slug: "hr_admin",
  permissions: ["employee.read", "employee.manage", "leave.approve"],
});

const users = await banata.users.listUsers({ limit: 20 });

const webhook = await banata.webhooks.createEndpoint({
  url: "https://example.com/api/webhooks",
  eventTypes: ["user.created"],
});
```

## Scope Rules

- the project-scoped API key is the normal scope boundary
- routine SDK calls should not require you to pass `projectId`
- one project's key should not be able to read or mutate another project's data

If you are building a customer app, prefer letting the API key determine scope.
