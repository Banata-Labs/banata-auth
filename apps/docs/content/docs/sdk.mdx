`@banata-auth/sdk` is the Banata admin and configuration client.

Use it when your backend needs to:

- manage users, orgs, invitations, SSO, SCIM, and RBAC
- manage API keys and webhooks
- update project configuration by code
- keep dashboard config and code-driven config in sync

The SDK talks to Banata over HTTP. It does **not** write to your app database directly.

```bash
npm install @banata-auth/sdk
```

---

## Initialization

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});
```

Start by creating the project and API key in the Banata dashboard. Then use that project-scoped key in the SDK.

This mirrors WorkOS-style usage: the dashboard bootstraps the project, then the SDK operates inside that project.

---

## What The SDK Controls

| Resource | Purpose |
|---|---|
| `configuration` | Dashboard-backed auth, branding, provider, and policy config |
| `users` / `userManagement` | User CRUD, sessions, banning, impersonation |
| `organizations` / `orgs` | Organizations, members, invitations |
| `sso` | Enterprise SSO connections |
| `directories` / `directorySync` | SCIM providers, users, and groups |
| `rbac` | Custom roles, permissions, and checks |
| `apiKeys` | API key lifecycle |
| `webhooks` | Webhook endpoints |
| `projects` | Project lifecycle |
| `vault` | Secret storage |
| `auditLogs` | Audit events |
| `emails` | Email providers, templates, and delivery tools |
| `domains` | Domain management |
| `portal` | Portal link generation |

## Dashboard + Code Stay In Sync

Banata is dashboard-first, but configuration can also be updated by code.

That means:

- dashboard changes affect the SDK immediately
- SDK config writes affect the dashboard immediately
- hosted UI reads the same project config as both of them

There is no separate "dashboard config" and "SDK config" store.

---

## Configure Auth Methods By Code

```ts
await banata.configuration.saveDashboardConfig({
  authMethods: {
    emailPassword: true,
    emailOtp: true,
    magicLink: false,
    passkey: false,
  },
  emailPassword: {
    requireEmailVerification: true,
    autoSignIn: true,
    minPasswordLength: 10,
    maxPasswordLength: 128,
  },
});
```

This updates the same project config the dashboard reads under:

- **Authentication -> Methods**
- **Authentication -> Providers** when you update provider credentials
- **Branding**
- the hosted auth UI

## Configure Branding By Code

```ts
await banata.configuration.saveBrandingConfig({
  primaryColor: "#0f766e",
  bgColor: "#f5f5f4",
  borderRadius: 14,
  font: "General Sans",
  logoUrl: "https://example.com/logo.svg",
});
```

Hosted auth pages and dashboard-backed rendering pick this up automatically.

---

## Configure Social Providers By Code

```ts
await banata.configuration.saveSocialProviderCredential({
  providerId: "github",
  clientId: process.env.GITHUB_CLIENT_ID!,
  clientSecret: process.env.GITHUB_CLIENT_SECRET!,
  enabled: true,
});
```

This mirrors what the dashboard does under **Authentication -> Providers**.

You can also inspect the current provider state:

```ts
const providers = await banata.configuration.getSocialProviderCredentials();
```

## Example Admin Workflow

```ts
const role = await banata.rbac.createRole({
  name: "HR Admin",
  slug: "hr_admin",
  permissions: ["employee.read", "employee.manage", "leave.approve"],
});

await banata.rbac.updateRole({
  id: role.id,
  permissions: ["employee.read", "employee.manage", "leave.approve", "payroll.read"],
});

const users = await banata.users.listUsers({ limit: 20 });

const webhook = await banata.webhooks.createEndpoint({
  url: "https://example.com/api/webhooks",
  eventTypes: ["user.created"],
});
```

## Error Handling

```ts
import { AuthenticationError, NotFoundError } from "@banata-auth/shared";

try {
  await banata.users.getUser("usr_123");
} catch (error) {
  if (error instanceof AuthenticationError) {
    console.error("Invalid API key");
  }

  if (error instanceof NotFoundError) {
    console.error("User not found");
  }
}
```

## Important Scope Rules

- The project-scoped API key is the normal scope boundary.
- `projectId` in SDK options is a legacy override, not the preferred managed path.
- Hosted Banata should not let one project's API key read or mutate another project's auth data.

If you are building a customer app, prefer letting the API key determine scope.
