## What Banata Auth Is

Banata Auth is auth infrastructure for:

- user management
- organizations
- SSO and directory sync
- RBAC
- audit logs
- webhooks
- API keys

There are **two distinct ways** to use it:

1. **Dashboard-first / API-first**
   Use the dashboard and `@banata-auth/sdk`. Banata is the system of record and your app talks to Banata over HTTP, similar to WorkOS.
2. **Self-hosted Convex runtime**
   Use `@banata-auth/convex`, `@banata-auth/nextjs`, and `@banata-auth/react` to run the auth engine inside your own Convex deployment.

Those are different product shapes. The self-hosted path is more like embedding the auth engine into your stack. The dashboard-first path is closer to a managed auth platform.

---

## Recommended Starting Point

If you want the **WorkOS-style developer experience**, start with:

- the Banata dashboard
- API keys
- `@banata-auth/sdk`
- thin app-side helpers such as `@banata-auth/nextjs`

If you need Banata to run inside your own infrastructure, use the self-hosted Convex guides instead.

## Package Roles

| Package | Primary Role |
|---|---|
| `@banata-auth/sdk` | Remote admin API client |
| `@banata-auth/nextjs` | Next.js proxy and server helpers |
| `@banata-auth/react` | React provider, hooks, and auth UI |
| `@banata-auth/convex` | Self-hosting kit for a local Convex deployment |
| `@banata-auth/shared` | Shared types, errors, and validation |

## Important Version Notes

- The current Convex integration is tested against `better-auth@1.4.x`.
- Published `@banata-auth/*@0.1.0` packages were released with broken `workspace:*` internal dependencies. The repo has the fix; until the patched release is published, consumers on `0.1.0` need overrides.
