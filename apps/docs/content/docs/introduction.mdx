## What is Banata Auth?

Banata Auth is an **open-source authentication infrastructure platform** — a drop-in replacement for [WorkOS](https://workos.com) that gives you everything you need to add enterprise-grade authentication to your application: **user management**, **organizations**, **SSO**, **directory sync**, **RBAC**, **audit logs**, **webhooks**, **API keys**, and more — all fully type-safe and MIT-licensed.

---

## Why Banata Auth?

### The Problem

Building authentication for modern SaaS applications means choosing between:

1. **Managed services** (WorkOS, Clerk, Auth0) — Fast to set up, but expensive at scale, vendor-locked, and you don't own your data.
2. **DIY from scratch** — Full control, but months of engineering work and ongoing security maintenance.
3. **Auth libraries** (NextAuth, Better Auth alone) — Good starting point, but missing enterprise features like SSO, SCIM, organizations, audit logs, and admin dashboards.

### The Solution

Banata Auth sits in the sweet spot: **enterprise-grade auth infrastructure that you own and control**. You get the developer experience of a managed service with the freedom of open source.

### Key Advantages

- **Open Source (MIT)** — No vendor lock-in. Fork it, modify it, self-host it. No per-user pricing, ever.
- **Enterprise-grade from Day One** — Organizations, RBAC, audit logs, webhooks, and MFA are built in, not premium add-ons.
- **Fully Type-safe** — TypeScript from SDK to database. Every API call, every hook, every configuration option is typed.
- **Real-time Auth State** — Built on Convex, so auth state changes propagate to clients instantly. No polling, no stale data.
- **Admin Dashboard Included** — A full-featured dashboard for managing users, organizations, SSO connections, API keys, branding, and more.
- **12 Authentication Methods** — Email/password, social OAuth (10 providers), magic links, email OTP, passkeys, anonymous auth, and more.

## How It Works

Banata Auth integrates into your Next.js + Convex application through a set of npm packages. You install them, configure your auth settings, and you're ready to go.

```
Your Next.js App
    │
    ├── @banata-auth/react    → React hooks & UI components
    ├── @banata-auth/nextjs   → Route handler & middleware
    └── @banata-auth/convex   → Backend auth engine (runs in Convex)
```

The auth engine runs entirely inside **Convex serverless functions**, giving you real-time subscriptions, automatic scaling, and zero cold starts. A reverse proxy in your Next.js app keeps auth cookies on your domain and handles OAuth callbacks.

> For a deep dive into the architecture, see [Self-Hosting](/docs/self-hosting).

## Authentication Methods

Banata Auth supports **12 authentication methods**, all configurable via the `BanataAuthConfig` object:

| Method | Description |
|---|---|
| Email & Password | Traditional signup with email verification |
| Social OAuth | Google, GitHub, Apple, Microsoft, Facebook, Twitter, Discord, Spotify, Twitch, LinkedIn |
| Magic Links | Passwordless sign-in via email link |
| Email OTP | One-time password sent via email |
| Two-Factor (TOTP) | Authenticator app codes with backup codes |
| Passkeys | WebAuthn biometric/hardware key authentication |
| Anonymous | Start without credentials, link account later |
| Multi-Session | Multiple concurrent sessions per user |
| API Keys | Programmatic access with scoped permissions |

## Security

Banata Auth implements defense-in-depth security:

- **Session Management** — Server-side sessions stored in Convex. Access tokens expire after 15 minutes; sessions expire after 7 days.
- **Password Requirements** — Configurable via Zod schema. Default: 8-128 characters.
- **HMAC-SHA256 Webhooks** — Webhook payloads are signed and verified with timing-safe comparison.
- **Rate Limiting** — Built-in defaults: 600 requests/minute general, 30/minute for sign-in, 10/minute for sign-up.
- **Input Validation** — 16 Zod schemas validate all input at the boundary.
- **Error Hierarchy** — 7 typed error classes with appropriate HTTP status codes prevent information leakage.

## How It Compares

### vs. WorkOS

WorkOS is the primary inspiration for Banata Auth. Both provide authentication infrastructure with enterprise features. The key differences:

- **Open source vs. proprietary** — Banata Auth is MIT-licensed. WorkOS is closed-source with per-user pricing.
- **Self-hosted vs. managed** — Banata Auth runs on your Convex deployment. WorkOS runs on their servers.
- **Real-time vs. request-response** — Banata Auth leverages Convex's real-time subscriptions. WorkOS uses traditional REST APIs.
- **React components included** — Banata Auth ships pre-built auth UI. WorkOS provides a headless API only.

### vs. Clerk

Clerk excels at developer experience with polished pre-built UI components. Banata Auth differs in:

- **Ownership** — You own your auth data and can self-host. Clerk is fully managed.
- **Audit logs** — Built-in with 30 auto-tracked events. Clerk offers audit logs only on enterprise plans.
- **Pricing** — Banata Auth is free forever. Clerk charges per monthly active user.

### vs. Auth0

Auth0 is the incumbent enterprise auth platform. Banata Auth differs in:

- **Simplicity** — Banata Auth is a focused TypeScript library, not a sprawling platform with its own scripting language.
- **Modern stack** — Built for the React/Next.js ecosystem from day one.
- **Cost** — Auth0's enterprise features require expensive tiers. Banata Auth includes everything.
- **Type safety** — Banata Auth is TypeScript-first with full type coverage.
