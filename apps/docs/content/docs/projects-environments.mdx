Projects are the foundation of isolation in Banata.

A project is the auth tenant your app connects to with an API key. Users, organizations, providers, roles, emails, branding, and enterprise connections all live inside that project boundary.

---

## What A Project Is

A project is not just a label in the sidebar. It is the scope boundary for:

- users
- sessions
- social accounts
- organizations and invitations
- roles, permissions, and resource types
- auth methods and provider credentials
- branding and redirects
- email templates and email delivery settings
- API keys, webhooks, audit logs, SSO, and SCIM

If two apps use two different projects, their auth data should stay isolated from each other.

---

## What The Default Project Actually Means

On the first dashboard load, Banata calls `ensure-default` for the signed-in dashboard admin.

If that admin has no accessible projects yet, Banata creates:

1. a first project for that admin
2. a `super_admin` role seeded with built-in permissions

This is the important clarification:

- the default project is **not** created per application end user
- the default project is **not** one global project shared by every Banata customer
- the default project is the first project created for a newly bootstrapped dashboard account or workspace context

In the current implementation, that first project is personalized from the dashboard admin identity. Older docs implied a single global `default` project. That was incorrect.

---

## Why The Project Switcher Exists

The project switcher exists because nearly every management surface in Banata is project-scoped.

When you switch projects in the dashboard, you switch which auth tenant you are managing. That affects:

- the users list
- the organization list
- the current roles and permissions
- provider credentials
- email templates
- branding
- audit logs
- enterprise connections

The project switcher is not decorative. It is the scope boundary for the dashboard itself.

---

## How App Scope Works

For a customer app, the normal scope rules are:

1. the app uses a project-scoped API key
2. Banata resolves the project from that API key
3. the app works inside that project automatically

You should not need to pass `projectId` around in routine app code. The API key is the main project-binding mechanism.

---

## What Is Shared Across Projects

The main thing shared across projects is the dashboard admin identity: the person who signs in to the Banata dashboard and manages projects.

The auth data served to customer apps should remain project-scoped.

---

## What To Verify In Production

When you are testing project isolation, verify all of these:

1. an API key from project A cannot list or mutate users from project B
2. users created in project A do not appear in project B
3. organizations created in project A do not appear in project B
4. provider credentials and email templates stay scoped to their own project
5. audit logs and webhooks remain project-scoped

If any of those fail, the project boundary is broken.

---

## Programmatic Access

The SDK exposes project lifecycle operations, but the normal product flow is still dashboard-first:

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});

const projects = await banata.projects.listProjects();
```

With a project-scoped API key, this returns the project attached to that key. Broader project lifecycle operations belong to dashboard and platform administration flows.
