Hosted sign-in is the default Banata experience.

Your app does **not** render the primary sign-in form itself. Instead:

1. your app links to Banata's hosted UI
2. Banata authenticates the user on the Banata domain
3. Banata redirects back to your app's `/auth/callback`
4. your app finalizes the session cookie on **your** domain

This is the closest Banata flow to WorkOS-style hosted auth.

---

## Required Inputs

Your hosted sign-in URL needs:

- `client_id`
- `redirect_url`

Example:

```ts
const url = new URL("/sign-in", process.env.BANATA_HOSTED_UI_URL!);
url.searchParams.set("client_id", process.env.NEXT_PUBLIC_BANATA_CLIENT_ID!);
url.searchParams.set("redirect_url", `${process.env.NEXT_PUBLIC_SITE_URL}/`);
```

`client_id` identifies the project for public config lookup.

`redirect_url` tells Banata where the user should end up after the hosted flow completes.

---

## What Must Be Configured First

Hosted UI only reflects what the dashboard has already enabled for that project.

Before you test it, make sure the project has:

1. at least one enabled auth method
2. provider credentials for any social method you expect to render
3. email delivery configured for email-based flows
4. branding configured if you want the hosted UI to match your project theme

---

## Customer App Requirements

Your app still needs:

1. `/api/auth/[...all]` proxy route
2. `/auth/callback` page
3. a server-side project API key

If your app is missing those pieces, the hosted UI may authenticate successfully on Banata but still fail to create a usable session on your app domain.

See [Quick Start](/docs/quickstart) and [Next.js](/docs/nextjs).

---

## Branding

Hosted sign-in pulls branding from the same dashboard-backed project config used elsewhere in Banata.

That means:

- dashboard branding changes update the hosted UI
- SDK branding changes update the hosted UI too
- there is no separate "hosted UI theme file" to maintain in your app

---

## Programmatic Updates

You can update hosted-auth behavior by code with the SDK:

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});

await banata.configuration.saveDashboardConfig({
  authMethods: {
    emailPassword: true,
    emailOtp: true,
    magicLink: true,
  },
  emailPassword: {
    requireEmailVerification: true,
    autoSignInAfterVerification: true,
  },
});
```

These updates are visible in the dashboard because the SDK writes to the same project config.

## Troubleshooting

### The hosted page loads but shows no buttons or form

The project is missing dashboard configuration. Check **Authentication -> Methods** and **Authentication -> Providers**.

### The hosted flow finishes but the user is still unauthenticated in the app

Check:

1. `/auth/callback` exists
2. `OPTIONS` is exported from your auth route
3. your auth route allows the hosted UI origin
4. your app uses a real project-scoped API key

### The hosted page does not match my app branding

Update the project branding in the dashboard or through `banata.configuration.saveBrandingConfig(...)`.
