API keys are the primary binding mechanism between your app and Banata.

The normal flow is:

1. create a project in the Banata dashboard
2. create a project-scoped API key inside that project
3. store the key on your server
4. let Banata resolve the project from that key

Your app should not need direct database access to use Banata.

---

## Why API Keys Matter

The API key does two jobs:

- it authenticates your app to Banata
- it tells Banata which project the app belongs to

That is why the product contract is dashboard-first and API-key-first. No usable customer integration should come online before a project and project-scoped API key exist.

---

## Create A Key In The Dashboard

For the normal product path:

1. sign in to the Banata dashboard
2. switch to the target project
3. open **API Keys**
4. create a new key
5. copy it immediately and store it securely

The raw key is returned once. It cannot be fetched later.

## Use The Key In Code

### Next.js server helpers

```ts
import { createBanataAuthServer } from "@banata-auth/nextjs/server";

export const { handler } = createBanataAuthServer({
  convexUrl: process.env.NEXT_PUBLIC_CONVEX_URL!,
  convexSiteUrl: process.env.NEXT_PUBLIC_CONVEX_SITE_URL!,
  apiKey: process.env.BANATA_API_KEY!,
});
```

### SDK

```ts
import { BanataAuth } from "@banata-auth/sdk";

const banata = new BanataAuth({
  apiKey: process.env.BANATA_API_KEY!,
  baseUrl: "https://auth.banata.dev",
});
```

In both cases, the key is what binds the app or server process to a project.

---

## Scope Rules

With the normal managed-service flow:

- one API key belongs to one project
- requests from that key should stay inside that project
- you should not need to pass `projectId` for routine usage

If a key from project A can read or mutate data from project B, that is a project-isolation bug.

---

## Security Model

Banata stores a hash of the key, not the raw key itself.

That means:

- lost keys cannot be retrieved
- compromised keys should be revoked and replaced
- raw keys should be stored in environment variables or a secrets manager

---

## Rotate And Revoke

Use separate keys for separate integrations. That lets you revoke one surface without breaking everything else.

Typical pattern:

1. create a new key
2. deploy the new key
3. verify traffic
4. revoke the old key

---

## Troubleshooting

### Requests authenticate, but hit the wrong project

Check that the app is using the correct project-scoped key.

### The dashboard works, but the app does not

Check:

1. `BANATA_API_KEY` exists on the app server
2. the app is using the new key you created
3. the key belongs to the expected project

### A key works locally but not in production

Check:

1. the production deployment actually has the new key
2. the production Banata backend was redeployed
3. the requests are hitting the intended Banata instance
